# Math Report Android Contract

This document defines the JSON boundary between the Python math-core diagnostics and the Android Research panel.

The contract is intentionally offline:

- Python generates reports during research/build/release preparation.
- Android reads bundled JSON assets or manually imported JSON.
- Android does not execute Python.
- Android does not contact a server or cloud API for diagnostics.
- The report is diagnostic-only and must not be presented as production cryptography.

## Contract version

Current contract: `kraken.math.curve_diagnostic.android.v1`

Android should reject or clearly mark reports with an unsupported `report_version`.

## Required fields

| Field | Type | Meaning |
| --- | --- | --- |
| `report_version` | string | Contract identifier. |
| `generated_at` | string | UTC ISO-8601 timestamp from the report generator. |
| `git_commit` | string or null | Source commit used to generate the report, if available. |
| `source.generator` | string | Generator package/tool name. |
| `curve.model` | string | Curve model. For now: `short_weierstrass_q`. |
| `curve.equation` | string | Human-readable equation. |
| `curve.a` | string | Exact rational coefficient. |
| `curve.b` | string | Exact rational coefficient. |
| `invariants.discriminant` | string | Exact discriminant value, duplicated for compact invariant display. |
| `invariants.singular` | boolean | Whether the curve is singular. |
| `invariants.j_invariant` | string or null | Exact j-invariant or null for unsupported/singular cases. |
| `diagnostics.discriminant` | string | Exact discriminant value. |
| `diagnostics.singular` | boolean | Whether the curve is singular. |
| `diagnostics.j_invariant` | string or null | Exact j-invariant or null for unsupported/singular cases. |
| `diagnostics.two_torsion.summary` | string | Human-readable 2-torsion summary. |
| `diagnostics.two_torsion.points` | array | Rational 2-torsion points found by exact diagnostics. |
| `diagnostics.lutz_nagell.summary` | string | Must state candidate/diagnostic scope. |
| `diagnostics.lutz_nagell.candidate_count` | integer | Number of candidate points. |
| `diagnostics.lutz_nagell.candidates` | array | Candidate points. |
| `diagnostics.torsion_probe_results` | array | Bounded scalar probing results. |
| `benchmark.runtime_ms` | number or null | Runtime for the diagnostic run if measured. |
| `warnings` | array of strings | Non-fatal warnings. |
| `unsupported_cases` | array of strings | Explicit unsupported cases. |
| `ui_wording.security_note` | string | Safe UI wording. |

## Relation to current Python report

The current Python `CurveDiagnosticReport.to_json_value()` emits the core diagnostic fields:

- `curve_equation`
- `a`
- `b`
- `discriminant`
- `singular`
- `j_invariant`
- `two_torsion_summary`
- `two_torsion_points`
- `lutz_nagell_candidates`
- `probed_torsion_orders`
- `warnings`
- `unsupported_cases`

For Android, a thin export adapter should wrap those fields into the Android contract above. The adapter should not change mathematical meaning; it only adds metadata, grouping, benchmark timing, and safe UI wording.

## Android display rules

The Research panel may show:

- curve equation;
- singular/nonsingular status;
- discriminant;
- j-invariant;
- 2-torsion summary and points;
- Lutz-Nagell candidate count;
- bounded torsion probe results;
- warnings and unsupported cases;
- benchmark timing;
- a diagnostic-only security note.

The Research panel must not say:

- this encrypts Kraken messages;
- this proves production security;
- this replaces reviewed cryptographic libraries;
- this improves cryptography without measured evidence;
- unsupported cases are solved.

## Versioning rules

- Additive fields are allowed in `v1` if Android ignores unknown keys.
- Removing or renaming fields requires a new `report_version`.
- Mathematical fields must use strings for exact rationals.
- Floating-point values are allowed only for benchmark timing and UI display metrics, not for mathematical truth.
